# QA Prompt Templates

下面这些 prompt 可以直接复制到业务项目里的 Codex 会话中使用。

## 1. 提测分诊

```text
你现在是测试负责人助手。
请读取当前分支的改动、相关源码和目录结构，输出本次提测的测试分诊报告。

要求：
1. 说明改动摘要
2. 说明影响模块和原因
3. 如果是客户端仓库，说明影响平台和原因；如果是后端仓库，说明影响调用方和原因
4. 给出风险等级：P0 / P1 / P2
5. 给出最小必测项
6. 给出建议补自动化项
7. 明确哪些判断来自源码证据，哪些属于推断

输出要短、清晰、可执行，默认中文。
```

## 2. 最小回归矩阵

```text
请先判断当前仓库是 Android、iOS、Windows/macOS、Web、后端，还是共享内核仓库。
然后只基于“当前仓库”的直接影响生成最小回归矩阵。

如果你认为兄弟平台仓库也需要跟进，请单独列一个“兄弟仓库 follow-up”区域，并说明证据，不要默认全部带上。

要求：
1. 按 P0 冒烟、P1 重点回归、P2 延后验证分组
2. 区分当前仓库直接风险和兄弟仓库 follow-up 风险
3. 不要全量铺开，只保留本次最值得测的内容
4. 每一项后面补一句为什么要测
```

## 3. 自动化补点

```text
请从测试负责人的角度，基于当前改动和现有测试资产，找出最值得补自动化的 3 到 5 个点。

要求：
1. 优先考虑高频、稳定、跨产品复用的能力
2. 每个点说明收益、实现难度、建议优先级
3. 标明适合 API 契约测试、集成测试、Web E2E、桌面自动化还是移动端黑盒自动化
```

## 4. 发版判断

```text
请基于当前代码改动、测试结果、已知问题和平台影响，输出发版判断报告。

要求：
1. 结论只允许是 可发布 / 有条件发布 / 阻塞发布
2. 如果不是 可发布，必须说明阻塞项
3. 列出已知风险和建议放行条件
4. 不要泛泛而谈，要结合本次改动
```

## 5. 后端仓库提测分诊

```text
当前仓库是后端仓库。
请基于当前 diff、接口定义、数据库相关代码、异步任务和配置变更，输出后端提测分诊报告。

要求：
1. 先判断当前仓库属于 API 服务、管理后台、任务调度、网关，还是公共服务
2. 列出受影响接口、数据表、异步任务、配置项
3. 列出受影响调用方：Android / iOS / PC / macOS / Web / 内部服务
4. 说明兼容性风险、数据一致性风险、回滚风险
5. 给出最小回归范围
6. 给出建议补自动化项
7. 明确哪些是源码证据，哪些是推断
```

## 6. 后端接口契约影响分析

```text
请分析这次后端改动对接口契约的影响，重点检查：

1. 请求参数是否变化
2. 返回字段是否变化
3. 字段含义、默认值、可空性是否变化
4. 错误码、鉴权、限流逻辑是否变化
5. 是否会影响旧客户端或旧版本调用方

输出格式：
- 直接受影响接口
- 潜在受影响调用方
- 必测兼容性场景
- 可延后场景
- 是否建议阻塞发版
```

## 7. 数据库和异步任务风险分析

```text
请从测试负责人的角度分析本次后端改动里的数据库和异步任务风险。

要求重点检查：
1. schema 变更
2. 索引和查询路径变化
3. 数据迁移脚本
4. 消息队列、定时任务、回调链路
5. 幂等性、重试、重复消费、脏数据风险

输出：
- 高风险点
- 最小验证步骤
- 建议监控项
- 是否适合自动化补点
```

## 8. 缺陷回归缩圈

```text
这是一个线上或提测缺陷。
请阅读相关模块源码，判断这个问题最可能影响的功能范围，并输出回归缩圈建议。

要求：
1. 先说根因候选
2. 再说同类风险点
3. 再说最小回归范围
4. 给出是否值得补自动化和理由
```

## 9. 需求转测试设计

```text
请把这个需求转成测试设计，但不要写冗长全量用例。

要求：
1. 先提炼 capability eval
2. 再提炼 regression eval
3. 标出 P0 / P1 / P2
4. 标出哪些适合开发自测，哪些必须由测试验证
5. 标出哪些适合自动化
```
